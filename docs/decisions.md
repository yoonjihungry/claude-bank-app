## 2026-07-20 — 버튼 커서는 컴포넌트마다가 아니라 base 레이어에서 전역으로 되돌린다

**배경**
`/desk`의 지역 펼치기 버튼에 손가락 커서가 안 뜬다는 지적. 조사해보니 개별 버그가 아니라
**Tailwind v4 preflight의 사양 변경**이었다 — v4부터 `button`/`[role=button]`의 기본값이
`cursor: pointer`에서 `cursor: default`로 바뀐다. 저장소 전체에서 `cursor-pointer`를 쓴 곳은
`TransactionForm`의 label 하나뿐이라, 지적받은 버튼만이 아니라 **가계부 화면을 포함한 모든 버튼**이
같은 상태였다.

**대안**
- (a) 지적받은 버튼에만 `cursor-pointer` 추가
- (b) 클릭 가능한 컴포넌트마다 `cursor-pointer` 유틸리티를 붙여 나가기
- (c) base 레이어에서 v3 동작으로 되돌리기(전역)

**결정 이유**
**(c) 전역 복구.** 증상이 한 버튼이 아니라 전 화면에 걸쳐 있으므로 (a)는 같은 지적을 계속 받게 된다.
(b)는 지금 있는 버튼을 다 고쳐도 **앞으로 추가되는 버튼마다 빠뜨릴 여지가 남는다** — 기본값이
어긋나 있는 문제는 기본값에서 고치는 게 맞다. base 레이어에 두면 유틸리티(`cursor-default`·
`cursor-grab`)가 항상 이기므로, 의도적으로 커서를 바꿔둔 곳(`LoginSheet`·`HeaderAuth`의 배경 클릭
영역, `/desk` 캐러셀 트랙)은 영향을 받지 않는다. 비활성 버튼은 `not-allowed`로 따로 구분했다.

이 규칙은 `/desk`(데모)만이 아니라 **가계부 본 서비스에도 적용된다.** `--desk-*` 토큰처럼
네임스페이스를 나누지 않은 이유는, 이건 도메인 스타일이 아니라 브라우저 기본 동작 복구이기 때문이다.

**교훈**: "이 버튼이 이상하다"는 제보를 그 버튼의 문제로 좁히지 말 것. 프레임워크 메이저 업그레이드
뒤의 이질감은 개별 버그가 아니라 **preflight/기본값 변경**인 경우가 있고, 그때는 범위를 먼저 재야 한다.


## 2026-07-16 — dev 서버: Next 버전은 유지하고 Turbopack만 끈다 + 포트 3100 고정

**배경**
로컬 서버를 띄우면 컴퓨터 전체가 멈출 듯 느려지는데 **유독 이 프로젝트만** 그렇다는 제보. 같은 머신의 다른 Next 프로젝트(Next 14 + webpack)는 개발 서버를 3개 동시에 띄워도 멀쩡하다. 조사에서 두 가지가 나왔다. (1) **확실한 것** — `npm run dev`를 중지해도 `next dev` 본체가 고아로 살아남는다(본체 848MB, 트리 합계 약 1GB). 껐다 켤 때마다 쌓여 램이 고갈된다. (2) **미확정** — node 프로세스가 초당 40~64개씩 2분 넘게 생성되는 폭주(누적 1,665개, 각 0MB = 뜨자마자 죽는 크래시 루프)가 있었고 동시에 `Thread failed to start`로 자원 고갈이 확인됐으나, 원인이 Turbopack인지 npx로 뜨는 figma MCP의 재시작 루프인지 가리지 못했다. Turbopack이 `internal error`로 캐시를 스스로 삭제한 기록은 남아 있다.

**대안**
- 번들러: (a) Next를 14/15로 다운그레이드 / (b) `--webpack`으로 Turbopack만 끄기 / (c) 그대로 두고 원인부터 확정
- figma MCP: (a) 제거 / (b) 유지
- 포트: (a) 기본 3000 유지 / (b) 3100 고정

**결정 이유**
번들러는 **(b) `--webpack`**. 사용자가 먼저 제안한 다운그레이드(a)는 기각했다 — 이 프로젝트는 React 19·Tailwind v4·Prisma 7이 Next 16에 얽혀 있어 버전을 내리면 줄줄이 딸려 내려온다. **무엇보다 Turbopack이 범인이라고 확정되지도 않았는데 미확정 가설 때문에 스택 전체를 되돌리는 건 순서가 뒤바뀐다.** `--webpack`은 같은 정보를 한 줄로 얻고 플래그만 빼면 즉시 되돌아간다. 실측으로 기동도 4.4s → 345ms로 빨라졌다. (c)는 그동안 작업을 못 하므로 기각 — 안전하게 일할 길을 먼저 열고 원인은 재발 시 잡는다.

figma MCP는 **(b) 유지**. MCP는 Claude Code 안에서만 뜨고 브라우저로 로컬에 접속할 때는 관여하지 않으므로, 제거해도 이 증상은 그대로다. 폭주 원인 후보로는 남아 있으나 근거 없이 지울 이유는 없다.

포트는 **(b) 3100 고정**. 기본 3000은 이 머신의 다른 프로젝트가 점유해 Next가 **에러 없이 조용히 3001로 밀어버린다** — `localhost:3000`을 열면 다른 앱이 뜨는데 눈치채기 어렵다. `dev`/`dev:turbo`/`start` 셋 다 `-p 3100`으로 맞췄다. 단 3100이 점유돼 있으면 똑같이 3101로 밀리고, 3100을 점유하는 가장 흔한 범인은 위의 고아 서버다 — 포트 고정만으로는 부족하고 고아 정리가 한 세트다(`predev` 자동 정리는 보류, 미도입).

**교훈**: "느려진다"의 원인을 램 총량으로 지목했다가 "다른 프로젝트 3개는 멀쩡하다"는 반례로 뒤집혔고, OneDrive 동기화 가설도 경로를 확인해 기각했다. 추측을 쌓지 말고 **재현·계측으로 좁혀야** 했다. 실제 원인(고아 프로세스)은 종료 명령이 성공했다고 응답한 직후 848MB짜리가 살아 있는 것을 확인하면서 잡혔다 — **도구가 "중지 성공"이라 해도 프로세스가 죽었다는 뜻은 아니다.**


## 2026-07-15 — 데스크 캐러셀도 1장씩 드래그: 드래그 로직 공통화와 "위치 우선" 인덱싱

**배경**
아티클에 넣은 "한 번 끌면 딱 1장"을 데스크 캐러셀(인기 데스크·맞춤 추천)에도 달라는 요구. 참고 사이트(innogridhr)는 react-slick이고 DOM에서 `slidesToScroll: 1`·`infinite: false`(prev 비활성)가 확인됐다 — 스와이프 한 번에 정확히 1장, 무한 순환 없음. 아티클과 달리 데스크는 센터 모드가 아니라 왼쪽 정렬이고, 카드가 8장이라 끝까지 1,624px인데 step은 252px이라 **나눠떨어지지 않는다**(6.44칸).

**대안**
- 코드 공유: (a) 두 캐러셀에 드래그 로직을 각각 복제 / (b) 공통 훅으로 추출 / (c) 한 컴포넌트로 통합
- 끝 처리: (a) 무한 순환(아티클처럼) / (b) 양끝 고정
- 나눠떨어지지 않는 끝: (a) `-index*step`으로 계산하고 결과만 `[-max,0]`으로 클램프 / (b) 인덱스의 확정 위치를 `min(i*step, max)`로 먼저 정하고 드래그를 얹기 / (c) 마지막 칸을 버려서 step 배수로만 이동

**결정 이유**
공통 훅(b) — `useOneStepDrag`. "효과만 동일하게"가 요구였으니 손맛이 갈라지면 안 되는데, 복제하면 한쪽만 고치는 사고가 난다. 다만 인덱스 해석(무한 순환이냐 양끝 고정이냐)은 훅이 정하지 않고 방향(`dir`)만 넘겨 호출측이 정하게 했다 — 이 부분은 캐러셀마다 진짜로 다르다. 한 컴포넌트 통합(c)은 센터/왼쪽정렬·순환/고정·페이징 유무가 전부 달라 분기만 늘어난다.

끝 처리는 양끝 고정(b). 참고 사이트도 `infinite: false`고 기존 네이티브 스크롤 동작도 그랬다. 무한 순환은 복제본이 필요해 데이터가 8장→24장이 되는데 얻는 게 없다.

나눠떨어지지 않는 끝은 **(b) 위치 우선**. 처음엔 (a)로 짰다가 실제 버그가 났다: 마지막 인덱스의 계산값(−1764)과 클램프된 실제 위치(−1624)가 어긋나, 되돌릴 때 **140px을 끌어도 꿈쩍 않는 死구간**이 생겼다. 계산값과 실제 위치가 다른 순간 드래그는 그 차이만큼 헛돈다. 인덱스→위치를 먼저 확정하면 이 어긋남 자체가 없어진다. (c)는 마지막 카드가 오른쪽 여백(`px-5`) 없이 잘려서 기각. 대신 마지막 칸만 112px로 짧게 이동하는데, 이건 결함이 아니라 끝에 정확히 붙기 위한 의도다.

**교훈**: 이 버그는 타입 체크·서버 컴파일·DOM 렌더 확인을 모두 통과했다. 인덱스 기반 이동에서 "칸 수 × 보폭"이 전체 거리와 나눠떨어지지 않으면 끝에서 반드시 어긋나므로, 경계값을 숫자로 시뮬레이션해 확인한다.


## 2026-07-15 — /desk 캐러셀 두 갈래(자동 배너 / 수동 아티클)와 네이티브 스크롤 폐기

**배경**
`/desk`에는 성격이 다른 가로 캐러셀이 둘 있다. 메인·프로모 배너는 자동으로 넘어가야 하고, 아티클 슬라이드는 첫 렌더부터 센터 모드(좌우 이웃이 걸쳐 보임)로 시작해 드래그로만 움직이되 **한 번 끌면 아무리 길게 끌어도 딱 1장만** 넘어가야 한다(참고 시안: Swiper `loop`+`slidesPerView:'auto'`+`centeredSlides`, 단 활성 슬라이드 확대는 제외). 둘 다 끝이 없어야 한다(무한 순환).

**대안**
- 구현 방식: (a) 두 캐러셀을 한 컴포넌트로 통합 / (b) 자동 배너와 아티클을 분리
- 아티클 이동: (a) 네이티브 스크롤 + CSS `scroll-snap` / (b) 네이티브 스크롤 + 스크롤 정지 후 JS 정렬 / (c) `translateX` + 포인터 드래그 직접 구현 / (d) Swiper 도입
- 페이징(`1/N`): (a) 슬라이드마다 렌더 / (b) 캐러셀 위 고정 오버레이 1개

**결정 이유**
컴포넌트는 분리(b), 아티클 이동은 `translateX` 직접 구현(c)을 택했다.

아티클에서 네이티브 스크롤 계열((a)·(b))은 **둘 다 시도했다 접었다**. `snap-mandatory`는 드래그 도중에도 카드를 강제로 끌어당겨 길게 끌면 눈에 띄게 떨렸고, 스냅을 걷어내고 스크롤이 멎은 뒤(120ms) JS로 정렬하는 방식도 관성과 JS가 같은 `scrollLeft`를 두고 경합하는 구조라 떨림이 남았다. 결정적으로 **"한 번에 1장"은 네이티브 스크롤로는 강제할 수 없다** — 관성 거리는 브라우저가 정하기 때문이다. 위치를 건드리는 주체를 JS 하나로 만들면 경합 자체가 사라지므로 `overflow-hidden` + `translateX`로 갔다. 자동 배너도 같은 방식이라 결과적으로 두 캐러셀의 이동 원리는 같아졌지만, 자동/수동과 페이징 유무로 요구가 갈려 컴포넌트는 계속 분리해 둔다.

Swiper 도입(d)은 보류했다. 필요한 건 loop+center+1장 스냅뿐이고 확대·내비게이션·autoplay는 안 쓰므로, 의존성 하나(약 40KB gz)를 더할 만큼의 이득이 없다고 봤다. 요구가 늘면 재고할 여지는 있다.

"1장 제한"은 판정만 막으면 부족하다 — 600px을 끌면 트랙이 600px 따라갔다가 1칸만 남기고 되돌아와 튕긴다. 그래서 **드래그 오프셋 자체를 1칸(`step`)으로 클램프**해 한 칸을 끌면 다음 카드가 정확히 중앙에 멈추게 했다. 임계값 40px 미만은 제자리 복귀.

무한 순환은 양쪽 다 복제본을 두고 콘텐츠가 픽셀 단위로 동일한 지점에서 순간 이동시켜 이음매를 감춘다(배너는 첫 슬라이드 1벌, 아티클은 3벌 중 가운데 벌 기준). 아티클의 복귀 시점은 `transitionend`가 아니라 **다음 `pointerdown`**이다 — 드래그로 정확히 한 칸을 채우면 손을 뗄 때 transform이 이미 목표값이라 `transitionend`가 오지 않아 복귀를 놓친다. `pointerdown`에선 어차피 애니메이션을 끄므로 순간 이동이 보이지 않는다.

페이징은 슬라이드마다 두면 이미지와 함께 밀려나므로 고정 오버레이 1개로 현재 인덱스만 표시한다.

## 2026-07-14 — /desk 데모 화면을 공통 셸 밖 독립 라우트로 + 별도 --desk-* 토큰

**배경**
피그마 시안 "Main_on"은 자체 상태바·헤더·하단 내비·푸터를 갖춘 완결형 모바일 화면이고, 팔레트(파랑 #006ffd·오렌지 #eb601c)도 가계부 본 서비스와 다르다. 이걸 기존 앱에 새 페이지로 얹을 방법과 색 처리 방침을 정해야 했다.

**대안**
- 렌더 위치: (a) 기존 `AppShell`(공통 헤더+탭바) 안에 넣기 / (b) 셸 밖 독립 전체화면
- 셸 우회 방법: (a) 루트 `layout.tsx`에서 AppShell 제거하고 기존 라우트를 `(app)` 라우트 그룹 레이아웃으로 이동 / (b) `AppShell`에 경로 예외(`BARE_ROUTES`) 추가

**결정 이유**
독립 전체화면(b)을 택했다. 시안이 자체 내비/푸터를 포함해 공통 셸과 겹치면 화면이 이중이 된다. 셸 우회는 라우트 그룹 재배치(기존 3개 페이지 이동)보다 영향 범위가 작은 `AppShell` 경로 예외(`BARE_ROUTES`)로 처리했다 — 기존 라우트 파일을 건드리지 않고 되돌리기도 쉽다. 색은 도메인이 달라 본 앱 의미 토큰(income/expense/primary)에 억지로 맞추면 의미가 어긋나므로, `--desk-*` 네임스페이스로 분리해 신설했다. 단, "색은 토큰만" 규칙은 유지해 화면 내 HEX/기본 팔레트 하드코딩은 두지 않았다. 이미지는 시안 도메인 사진이라 회색 플레이스홀더로 두고 실제 이식은 분리했다.

## 2026-07-08 — 고정거래 등록 진입점 + 자동 생성 멱등성 + 커스텀 색상 선택기

**배경**
로그인/DB(Phase 8)를 미룬 채 현재 localStorage 앱에 "매달 반복되는 고정지출/수입"을 추가하기로 했다. 어디서 등록할지, 자동 생성의 중복을 어떻게 막을지, 관리 UI를 어디에 둘지가 쟁점이었다. 더불어 카테고리 색을 직접 고르는 UX도 손봤다.

**대안**
- (등록 진입) 예산과 나란히 세그먼트 탭 vs 하단 탭 신설 vs **거래 입력 폼의 '매달 반복' 체크** vs 별도 관리 화면
- (자동 생성 중복 방지) 규칙에 `lastRunMonth` 하나만 기록 vs 생성된 거래 존재 여부로 판정 vs **`generatedMonths`(생성한 달 목록) + 결정적 거래 id**
- (색 선택) OS 기본 `input[type=color]` 유지 vs 고른 값을 매 이벤트마다 스와치 추가 vs **앱 내 커스텀 색상 선택기 + ✓로 확정**

**결정 이유**
고정거래는 "한 번 등록해두고 거의 안 건드리는" 설정성 항목이라, 등록은 사용자가 이미 "매달 나가는 돈"을 떠올리는 순간인 **거래 입력 폼의 '매달 반복' 체크**로 녹였다(세그먼트 탭은 예산과 화면을 경쟁시켜 기각). 관리(수정·해지)는 가끔만 필요하므로 카테고리 페이지 하단 **접힘 섹션**에 두어 평소 화면을 어지럽히지 않게 했다. 주기는 개인용 범위에서 **매월만** 지원한다.

자동 생성은 앱 마운트 시 1회 실행하되, 규칙에 **`generatedMonths`를 기록**하고 생성 거래 id를 `규칙id__YYYY-MM` **결정적 값**으로 만들어 멱등하게 했다 — 같은 달을 두 번 만들지 않고(React StrictMode 이중 실행에도 안전), 사용자가 자동 생성분을 지워도 그 달은 이미 '생성됨'이라 **되살아나지 않는다**(사용자 의도 존중). 폼에서 이번 달치를 직접 입력한 경우 그 달을 `generatedMonths`에 시드해 중복을 피한다. 규칙 삭제는 규칙만 지우고 이미 생성된 과거 거래는 실제 기록이라 남긴다. 필드는 모두 선택적/신규라 기존 localStorage 데이터와 호환된다.

색 선택기는 OS 기본 팝업이 **확인 버튼이 없고 값이 드래그마다 바뀌어** 혼란스러웠다(중간에 매 이벤트마다 스와치를 추가하는 방식도 스팸이라 기각). 그래서 **앱 내 커스텀 선택기**(채도/명도 박스 + 색조 슬라이더, HSV↔RGB↔HEX 변환은 `utils/color.ts`)를 만들고, 우상단 **✓(확인)** 을 눌러야 팔레트에 새 스와치로 확정·선택되게 했다 — "고르고 → 확인 → 목록에 추가"라는 명시적 흐름. 팔레트는 기본색 ∪ 사용 중인 색 ∪ 이번에 추가한 색으로 구성해, 한 번 쓴 커스텀 색은 다음에도 스와치로 재사용된다.


## 2026-07-07 — 인증·저장소 세부 결정 확정 (Prisma / Google / 게이팅 B / 로그인 진입)

마이그레이션 계획에서 "추후 확정"으로 미뤄뒀던 항목들을 진행 중 확정했다.

**ORM: Prisma (vs Drizzle)**
- 대안: Drizzle(가벼움·SQL 친화) vs Prisma(러닝커브 낮음·타입 안전·마이그레이션 도구 완비).
- 결정: 개인용·소규모라 번들 크기/성능보다 개발 편의가 우선이고, NextAuth 공식 어댑터(`@auth/prisma-adapter`)가 있어 인증 연동이 가장 매끄럽다. Prisma 7의 `prisma-client` 제너레이터 + 드라이버 어댑터(PrismaPg)로 Neon(서버리스 Postgres)에 연결한다.

**인증 Provider: Google (vs 이메일/비밀번호)**
- 대안: 이메일+비밀번호 직접 관리 vs OAuth(Google 등).
- 결정: 비밀번호 저장·재설정 부담이 없고 개인이 이미 쓰는 계정으로 바로 로그인되는 Google OAuth. 계정/세션 테이블은 NextAuth 모델(User/Account/Session)로 위임한다.

**로그인 게이팅: B — 비로그인 병행 (vs A 로그인 필수)**
- 대안: A 로그인 필수(게이트 화면) vs B 비로그인은 localStorage로 계속 쓰고 로그인 시 서버 동기화.
- 결정: B. 개인용이라 로그인 강제는 진입 마찰이 크고 기존 localStorage 사용 경험을 끊지 않는다. 로그인 강제 미들웨어가 없어 auth 설정도 단일 Node 런타임으로 단순해진다(edge 분리 불필요). 최초 로그인 시 로컬→서버 이관 프롬프트는 Phase 8에서 다룬다.

**로그인 진입: 즉시 리다이렉트 대신 안내 바텀시트**
- 대안: 로그인 버튼 → 곧바로 `signIn('google')`(구글로 즉시 이동) vs 안내 시트를 먼저 노출.
- 결정: 즉시 리다이렉트는 맥락 없이 구글 화면으로 튕겨 이탈·당황을 유발하고, 게이팅 B("로그인은 선택")와도 어긋난다. 헤더 클릭 시 **바텀시트**로 왜 로그인하는지(기기 간 데이터 유지)와 "나중에 하기"를 먼저 보여주고, 시트 안에서만 실제 로그인을 실행한다. 혜택 나열 섹션은 개인용 앱에 과하다고 보고 헤드라인+버튼 중심으로 간결화했다.

## 2026-07-06 — 신용카드 할부: 소비는 '구매월 전액', 카드값은 '청구월 분할'

**배경**
지출에 결제수단(현금/체크/신용)을 구분하고, 신용카드는 후불이라 별도로 집계하고 싶다는 요구가 있었다. 특히 할부는 "이번 달에 얼마 썼나(소비)"와 "이번 달에 카드로 얼마 빠져나가나(청구)"가 서로 다른 값이 되어, 두 관점을 어떻게 화면에 담을지 정해야 했다. (600,000원 6개월 할부를 예로 두고 논의)

**대안**
- (소비 집계) 구매월에 전액 반영 vs 매달 분할 반영 vs 개월수는 표시만 하고 집계는 전액
- (신용카드 총액) 소비금액 총합에 포함 vs 별도 분리
- (할부 데이터) 거래 1건 + 개월수 메타 vs 회차마다 거래 여러 건 생성

**결정 이유**
사용자 선택에 따라 **소비는 구매 시점, 카드값은 청구 시점** 두 기준을 병행한다. `이번달 소비금액`은 구매한 달에 **전액**을 반영하고(현금·체크 / 신용카드로 분해 표시), 신용카드도 소비 총합에는 포함한다 — "이번 달 얼마 썼나"는 산 시점 기준이 직관적이기 때문이다. 반면 `이번 달 카드 청구 예정` 카드는 **청구 시점** 기준으로, 할부는 이번 달 회차분(예: 3/7)만 합산하고 과거에 산 할부도 이번 달 회차가 있으면 잡는다 — 실제 카드값(현금흐름)을 보여주기 위해서다. 두 카드의 금액이 다른 것은 **의도된 차이**이며, 그 차이는 과거 구매 할부의 이번 달 회차분에서 생긴다.

할부는 **거래 1건 + `installmentMonths` 메타**로 저장한다(회차마다 거래를 만들면 원장이 지저분해지고 수정·삭제 정합성이 깨진다). 청구액은 원금 균등분할하되 나눠떨어지지 않는 **나머지는 1회차에 몰아** 합계가 총액과 일치하게 한다. 무이자·구매 당월 1회차 시작으로 단순화했다(이자·다음달 청구 규칙은 개인용 범위에서 과함). 결제수단/할부 필드는 모두 `Transaction`에 **선택적**으로 추가해, 필드가 없는 기존 localStorage 데이터도 '미지정/일시불'로 안전하게 동작한다(저장소 계층·리듀서 구조 변경 없음).

## 2026-07-03 — 카테고리를 상수에서 편집 가능한 영속 상태로 승격

**배경**
카테고리가 `constants/categories.ts`에 하드코딩돼 있어 사용자가 추가/수정/삭제할 수 없었다. 사용자는 예산 페이지에서 카테고리를 직접 관리(＋로 추가, 톱니바퀴로 수정·삭제)하길 원했고, 색상도 직접 지정하길 원했다.

- 카테고리를 LedgerContext 상태로 승격하고 localStorage에 영속화(기본값으로 시드)

- 그래프(이미지4): 신규 일자별 차트 신설 vs 기존 월별 막대차트 대체

**결정 이유**
대시보드는 **3섹션만** 남겨 "한눈에 보는 요약" 역할에 집중시키고, 분석성 요소(카테고리 파이·추이 차트·최근 내역)는 이미 폼/필터/목록이 있는 **거래 탭으로 이동**했다. 요약 카드는 섹션3(이번달 소비금액)과 의미가 겹쳐 삭제했다. 그래프는 기존 6개월 월별 막대(`MonthlyChart`)를 **이미지4 스타일의 일자별 순액 막대+누적선(`DailyTrendChart`)으로 대체**했다 — 사용자가 그 디자인을 원했고, 일 단위 흐름이 개인 가계부의 "이번 달 씀씀이 추세"를 더 잘 보여준다. 색은 참조의 보라 대신 프로젝트 포인트 컬러 `--primary`를 써 토큰 규칙을 지켰다(dataviz 검증 통과).

세부 결정 두 가지: (1) **섹션1(오늘의 소비)은 월 네비게이터와 분리**해 항상 실제 오늘을 가리킨다 — "오늘 얼마 썼나"는 조회 중인 월과 독립된 정보다. (2) **새로고침 아이콘은 기준 시각만 갱신**한다 — 서버가 없어 단일 탭에서 데이터는 항상 최신이므로 재조회 액션을 새로 만들 실익이 없다(저장소 계층/리듀서를 건드리지 않음). 주말색은 관례색(일 빨강·토 파랑)이라 의미 토큰과 섞지 않고 `--weekend-sun/sat` 토큰으로 분리했다.

## 2026-07-03 — 톤을 "밝은 핀테크"로 전환하고 다크 모드는 미지원

**배경**
초기 톤은 "신뢰감 있고 차분한" 쿨블루였는데, 사용자가 밝고 다채로운 핀테크(토스·카뱅 계열) 색감을 원했다. 어느 방향으로 갈지, 다크 모드까지 만들지 정해야 했다.

**대안**
- 차분한 기존 톤 유지
- 밝은 핀테크로 전환 — 후보 3종(A 밝은 블루 / B 웜 파스텔 그레이프 / C 프레시 민트 틸)을 미리보기로 비교
- 라이트+다크 양쪽 팔레트 구성 vs 라이트 전용

**결정 이유**
A안(밝은 핀테크·블루)을 선택했다. 사용자가 미리보기 비교에서 A를 골랐고, 특히 다크 모드용으로 잡아둔 **선명한 색값을 라이트 배경에 그대로 쓰길** 원해서 그 비비드 값을 라이트에 적용했다. 다크 모드는 명시적으로 안 만들기로 했다 — 개인용 앱이라 요구가 없고, 라이트만 유지하면 토큰이 `:root` 하나로 단순해진다. 비비드 색은 흰 배경 대비가 다소 낮다는 점(특히 금액 텍스트)을 알고도 사용자의 선호를 우선했고, 필요 시 텍스트용 색만 따로 딥하게 조정하기로 했다. 카테고리 8색은 dataviz 검증(ΔE 최소 24.2)을 통과한 팔레트를 썼다.

## 2026-07-02 — 색상 디자인 토큰(shadcn HSL) + Tailwind v4 매핑

**배경**
컴포넌트마다 `text-red-500` 같은 Tailwind 기본 색을 하드코딩하고 있어, 톤을 일관되게 잡고 이후 다크 모드로 확장하기 어려웠다. "신뢰감 있고 차분한 / 지출·경고는 눈에 띄게"라는 톤에 맞는 의미 기반 색 체계가 필요했다.

**대안**
- Tailwind 기본 팔레트(`blue-600`, `red-500` 등)를 계속 하드코딩
- `tailwind.config.js`(v3 방식)에 `hsl(var(--x))` 색을 매핑
- shadcn HSL 컨벤션 토큰을 `:root`에 두고 Tailwind v4의 `@theme inline`으로 매핑
- 색상 값을 hex로 직접 토큰화

**결정 이유**
shadcn HSL 컨벤션(`--token: "H S% L%"`) + Tailwind v4 `@theme inline` 매핑을 선택했다. 프로젝트가 이미 Tailwind v4(`@tailwindcss/vite`, config 파일 없음)라 v3용 `tailwind.config.js`를 새로 만드는 것은 방식이 어긋난다. HSL 채널 분리 표기는 다크 모드에서 명도(L)만 조정하기 쉽고, `hsl(var(--x) / a)` 형태의 알파 합성도 자연스럽다. 의미 기반 토큰(income/expense/warning/destructive)으로 두면 색을 한 곳에서 바꿔도 전역 반영된다. 팔레트는 눈대중이 아니라 dataviz 검증 스크립트로 확정했고(적록 색맹 구분 ΔE 18.3로 통과), 다크 모드는 당장 불필요해 `:root`만 우선 정의했다. 토큰 인프라만 먼저 만들고 컴포넌트 적용은 별도 단계로 분리했다.

## 2026-07-02 — 탭 기반 단일 페이지 네비게이션 (라우터 미도입)

**배경**
대시보드·거래·예산 세 화면을 오가야 하는데, 개발 초기에는 세 페이지가 한 화면에 세로로 나열돼 있어 탐색이 불편했다. 화면 전환 방식을 정해야 했다.

**대안**
- `react-router-dom` 도입 후 URL 라우팅(`/dashboard`, `/transactions`, `/budget`)
- `App.tsx`에서 `useState`로 활성 탭을 관리하는 단순 조건부 렌더링
- 기존처럼 한 페이지에 전부 세로로 나열

**결정 이유**
`useState` 기반 탭 전환을 선택했다. 화면이 3개뿐이고 서버·로그인 없이 브라우저에서 동작하는 개인용 SPA라 URL 라우팅/딥링크 요구가 없어, 라우터 의존성을 추가하는 비용이 이득보다 크다. 전부 나열하는 방식은 스크롤이 길고 맥락이 섞여 제외했다. 추후 딥링크나 뒤로가기 지원이 필요해지면 이 지점에 `react-router-dom`을 도입한다.
